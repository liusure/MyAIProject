# 项目宪章：AI 驱动的智能选课系统

## 核心原则

### 一、代码质量标准

1. **类型安全优先**
   - 所有函数必须包含完整的类型注解（Type Hints）
   - 使用 `mypy` 进行静态类型检查，通过率 100%
   - 复杂数据结构使用 `dataclass` 或 `Pydantic` 模型定义

2. **文档完整性**
   - 每个公共 API 必须有 Google 风格的 Docstring
   - 复杂算法逻辑必须包含注释说明其决策依据
   - 维护 `README.md` 和 `ARCHITECTURE.md` 文档

3. **模块化设计**
   - 严格分层：数据层 → 业务逻辑层 → API 层 → 表现层
   - 每个模块职责单一，圈复杂度 ≤ 10
   - 使用依赖注入，避免全局状态

### 二、测试标准

1. **单元测试覆盖率**
   - 核心业务逻辑覆盖率 ≥ 85%
   - 边界条件和异常路径必须有测试覆盖
   - 使用 `pytest` 框架，每个测试独立运行

2. **集成测试要求**
   - LLM API 调用必须有 Mock 测试
   - 数据库操作必须有事务回滚测试
   - 第三方服务集成必须有沙箱环境验证

3. **性能测试**
   - LLM 响应时间监控：P95 < 5秒
   - 选课算法必须在 2 秒内完成 1000 门课程的匹配
   - 压力测试：支持 100 并发用户

### 三、用户体验一致性

1. **响应式交互**
   - 所有异步操作必须提供加载状态提示
   - 错误信息必须友好且可操作（如"网络错误，点击重试"）
   - 选课结果必须有清晰的视觉反馈（成功/冲突/失败）

2. **对话式交互规范**
   - LLM 输出格式统一：JSON + 自然语言解释
   - 保持对话历史上下文（至少最近 10 轮）
   - 支持中断和回滚操作

3. **无障碍访问**
   - 支持键盘快捷键操作
   - 颜色对比度符合 WCAG 2.1 AA 级标准
   - 屏幕阅读器兼容

### 四、性能要求

1. **响应时间**
   - 首屏加载 < 1.5 秒
   - LLM 流式输出首字符 < 1 秒
   - 选课冲突检测 < 500ms

2. **资源效率**
   - 单用户内存占用 < 200MB
   - LLM Token 使用优化：缓存重复查询结果
   - 数据库查询索引覆盖率 100%

3. **可扩展性**
   - 支持水平扩展（无状态设计）
   - 课程数据支持增量更新
   - LLM 提供商可插拔（支持 MiMo/OpenAI/Claude 切换）

## 开发工作流

### 代码审查要求
- 每个 PR 至少 1 人审查
- 必须通过 CI/CD 检查（类型、测试、格式）
- 变更日志必须更新 CHANGELOG.md

### 版本控制规范
- 使用语义化版本（SemVer）
- 分支策略：main → develop → feature/*
- Commit 信息遵循 Conventional Commits

### 质量门禁
- 测试覆盖率下降 < 1%
- 无 Critical 级别的安全漏洞
- 性能基准测试无退化

## 项目特定约束

### 数据隐私
- 学生选课记录加密存储
- LLM 对话数据匿名化处理
- 支持 GDPR 删除权

### LLM 集成规范
- 所有 LLM 调用必须有降级方案（缓存/规则引擎）
- Token 使用量每日报告
- 敏感信息（API Key）禁止硬编码

### 选课算法要求
- 必须考虑课程冲突检测（时间/地点/先修课程）
- 支持优先级策略（学分/兴趣/专业要求）
- 提供选课成功率预估
