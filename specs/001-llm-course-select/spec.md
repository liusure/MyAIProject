# 功能规格：LLM 自动选课系统

**功能分支**：`001-llm-course-select`
**创建日期**：2026-04-22
**状态**：草案
**输入**：用户描述：LLM 驱动的智能选课系统

## 明确排除范围

- 系统**不执行**真实选课操作（不对接学校教务系统的选课接口）
- 实际选课操作由学生在学校选课系统中自行完成
- 系统仅提供课程推荐、冲突检测、方案生成与导出功能

## 用户场景与测试 *（必填）*

### 用户故事 1 - 自然语言选课推荐（优先级：P1）

作为学生，我希望用自然语言描述我的选课偏好（如"我想选周一到周三上午的计算机课程，不要和我已选的数据库课程冲突"），系统自动理解我的需求并推荐合适的课程方案。

**为什么是此优先级**：这是系统的核心价值——用自然语言交互替代手动浏览课程目录，直接决定用户是否愿意使用本系统。

**独立测试**：可以通过输入一条自然语言选课需求，验证系统是否返回匹配的课程推荐列表来独立测试。

**验收场景**：

1. **假设** 学生已登录系统，**当** 学生输入"我想选下学期的机器学习相关课程，周三下午有空"，**那么** 系统返回至少 3 个符合条件的机器学习课程方案
2. **假设** 学生输入了包含时间偏好的需求，**当** 系统处理请求时，**那么** 推荐的课程必须在学生指定的时间段内
3. **假设** 学生输入模糊需求（如"推荐一些有趣的选修课"），**当** 系统无法精确匹配时，**那么** 系统应请求更多信息或基于通用偏好推荐

### 用户故事 2 - 课程时间冲突检测（优先级：P1）

作为学生，我希望查看推荐课程的可视化时间表，系统自动标出时间、地点和先修课程冲突，让我在对照选课前就能发现问题。

**为什么是此优先级**：冲突检测是选课系统的基本要求，学生在选课时最常见的痛点就是事后发现课程冲突，必须在推荐阶段就解决。

**独立测试**：可以通过选择一组包含已知冲突的课程，验证系统是否正确标出所有冲突来独立测试。

**验收场景**：

1. **假设** 两门推荐课程在同一时间段，**当** 学生查看时间表时，**那么** 系统必须高亮标出冲突并提供替代方案
2. **假设** 推荐课程有先修课程要求，**当** 学生未满足先修条件时，**那么** 系统必须提示先修课程信息
3. **假设** 课程在不同校区但时间间隔不足，**当** 时间间隔 < 30 分钟时，**那么** 系统应提示通勤风险

### 用户故事 3 - 方案收藏与导出（优先级：P2）

作为学生，我希望在确认推荐方案后，能收藏方案并导出为 PDF/图片，以便在学校选课系统开放时快速对照操作。

**为什么是此优先级**：收藏和导出让学生能离线对照选课，是从"推荐"到"辅助选课"的关键衔接。但从功能完整性看，推荐和冲突检测必须先完成。

**独立测试**：可以通过收藏一个推荐方案并验证其持久化存储，以及导出为 PDF 后内容完整性来独立测试。

**验收场景**：

1. **假设** 学生确认了一个满意的推荐方案，**当** 学生点击"收藏方案"时，**那么** 系统将方案持久化存储，学生可在"我的方案"中查看
2. **假设** 学生需要离线对照选课，**当** 学生点击"导出 PDF"时，**那么** 系统生成包含课程名称、时间、地点的 PDF 文件
3. **假设** 学生有多个收藏方案，**当** 学生在"我的方案"页面时，**那么** 系统展示所有收藏方案并支持比较

### 用户故事 4 - 教务数据导入与规则配置（优先级：P3）

作为教务管理员，我希望能批量导入课程数据（时间、地点、容量、先修关系），并设置选课规则（如学分上限、选课时段），以确保推荐系统使用准确的数据。

**为什么是此优先级**：数据是推荐系统的基础，但此功能面向管理角色而非学生核心流程，因此优先级低于学生端功能。

**独立测试**：可以通过导入一批课程数据文件并验证系统是否正确解析和存储来独立测试。

**验收场景**：

1. **假设** 教务上传了 CSV 或 Excel 格式的课程数据文件，**当** 系统处理导入时，**那么** 系统自动识别文件格式并正确解析所有课程信息
2. **假设** 教务设置了选课学分上限为 25 学分，**当** 学生推荐方案超过此上限时，**那么** 系统提示学分超限
3. **假设** 导入的课程数据包含先修关系，**当** 学生后续选课时，**那么** 冲突检测应考虑先修课程约束

### 边缘场景

- LLM 服务不可用时如何降级处理（提供课程目录搜索替代方案）？
- 学生输入的语言包含非中文/非英文内容时如何处理？
- 导入的课程数据格式校验失败时如何反馈具体错误行？

## 需求 *（必填）*

### 功能需求

- **FR-01**：系统必须支持自然语言解析学生课程需求，理解时间偏好、课程类型、学分要求等意图
- **FR-02**：系统必须检测课程冲突，包括时间冲突、地点通勤冲突和先修课程冲突
- **FR-03**：系统必须为每次请求提供课程推荐方案；当匹配到 3 个及以上时展示全部，不足 3 个时仅展示实际匹配到的方案，不强制凑数
- **FR-04**：系统必须记录所有操作日志，包括用户操作时间、操作类型和操作结果
- **FR-05**：系统必须支持收藏推荐方案，并支持导出为 PDF 格式
- **FR-06**：系统必须支持教务管理员批量导入课程数据（支持 CSV 和 Excel 格式，自动识别）
- **FR-07**：系统必须支持教务管理员配置选课规则（学分上限、选课时段等）
- **FR-08**：当 LLM 服务不可用时，系统必须提供基于关键词搜索的降级方案
- **FR-08a**：前端在 LLM 处理期间展示简单 spinner 和"加载中"文字提示；搜索无结果时展示"无结果"提示
- **FR-05a**：学生收藏方案数量不设上限，"我的方案"页面按收藏时间倒序分页展示
- **FR-09**：系统必须通过 Session-based 认证区分学生和管理员角色
- **FR-10**：LLM 输出必须为 JSON 结构化格式，包含自然语言解释和结构化推荐数据

### 非功能需求

- **NFR-01**：LLM 响应时间 P95 < 5 秒
- **NFR-02**：支持 100 并发用户无性能退化
- **NFR-03**：选课高峰期可用性 99.9%
- **NFR-04**：前端界面采用现代、清爽的 UI 设计风格，主色调为蓝色系（推荐 #1a73e8 或相近色），整体视觉风格统一
- **NFR-05**：所有卡片类组件（课程卡片、方案卡片、推荐方案等）使用圆角设计（border-radius ≥ 8px），布局整齐、间距一致
- **NFR-06**：所有可交互元素（按钮、卡片、链接）具备流畅的悬停动画效果（transition 时长 200-300ms），提供视觉反馈
- **NFR-07**：数据安全通过 HTTPS 传输加密保障，数据库存储不额外加密

### 关键实体

- **学生（Student）**：系统用户，具有学号、已选课程列表、专业、年级等属性
- **课程（Course）**：课程编号、名称、学分、授课教师、上课时间、上课地点、课程容量、先修课程列表
- **收藏方案（SavedPlan）**：学生收藏的推荐课程方案，关联学生和课程列表，记录方案名称、收藏时间、导出状态
- **选课规则（SelectionRule）**：学分上限、选课开放/截止时间、选课优先级策略
- **对话记录（Conversation）**：学生与 LLM 的对话历史，用于上下文理解和操作审计

## 成功标准 *（必填）*

- **SC-001**：用户选课准备时间相比传统方式减少 70%
- **SC-002**：推荐方案满意度（学生愿意按方案选课）提升 40%
- **SC-003**：用户满意度评分 > 4.5/5
- **SC-004**：系统在选课高峰期（100 并发用户）保持正常响应，P95 响应时间 < 5 秒
- **SC-005**：课程冲突检测准确率 100%（无漏检冲突）
- **SC-006**：LLM 服务降级时，关键词搜索替代方案的推荐满意度 > 3.0/5

## 假设

- 学生使用现代浏览器（Chrome/Firefox/Safari 最近两个大版本）
- 课程数据格式遵循教务系统标准导出格式（CSV/Excel）
- LLM 服务（MiMo/OpenAI/Claude）可通过 API 稳定访问
- 选课高峰期预计每学期 2-3 次，每次持续 1-3 天
- 学生已通过学校统一身份认证登录系统
