# Quickstart: Multi-Step Course Recommendation

## 测试场景 1：正常两步推荐流程

**前置条件**: 学生已上传包含 2000+ 条课程记录的课表文件，涵盖计算机、数学、物理、文学等学科。

**操作**: 学生输入"推荐计算机课程"

**预期流程**:
1. 系统从 SessionStore 提取所有唯一 category 值（如 ["计算机科学与技术", "数学", "物理学", "文学", ...]）
2. 第一步 LLM 调用：发送用户需求 + 学科多选题 → 返回 ["计算机科学与技术"]
3. 本地过滤：从 2000+ 条中筛选出约 200 条计算机课程
4. 第二步 LLM 调用：发送过滤后的非敏感课程信息 + 用户需求 → 返回 1-3 个推荐方案
5. 本地回查：根据课程序号查找完整课程详情
6. 冲突检测：验证推荐方案中无时间冲突
7. 返回包含完整课程信息的推荐方案

**验证点**:
- 第一步 LLM 收到的消息中仅包含学科列表，无课程详情
- 第二步 LLM 收到的消息中不含 capacity、campus、semester 等敏感字段
- 最终推荐方案包含完整课程信息（名称、时间、地点、教师、学分）
- 推荐方案中无时间冲突

## 测试场景 2：学科过滤无匹配

**前置条件**: 学生上传了课表文件。

**操作**: 学生输入"推荐创意写作课程"（假设课表中无此学科）

**预期流程**:
1. 第一步 LLM 返回空学科列表
2. 系统回退到使用全部课程数据
3. 第二步在全部课程上执行推荐

**验证点**:
- 系统不报错，正常完成推荐流程
- 返回结果（可能为空或包含关键词相关的课程）

## 测试场景 3：LLM 调用失败降级

**前置条件**: MiMo API 不可用或触发内容过滤。

**操作**: 学生输入任意选课需求

**预期流程**:
1. 第一步 LLM 调用失败
2. 系统回退到使用全部课程数据
3. 第二步 LLM 调用可能也失败
4. 系统使用 FallbackLLMProvider 降级

**验证点**:
- 系统不崩溃，返回降级结果
- degraded=True 标记正确设置
- 降级结果不缓存

## 测试场景 4：第二步无匹配课程

**前置条件**: 学科过滤成功，但第二步 LLM 返回空推荐列表。

**预期流程**:
1. 第一步正常完成学科过滤
2. 第二步 LLM 返回空推荐列表
3. 系统返回"无匹配课程"提示

**验证点**:
- 不做额外降级（不回退到关键词搜索）
- 返回友好的中文提示信息
