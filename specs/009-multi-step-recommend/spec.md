# Feature Specification: Multi-Step Course Recommendation

**Feature Branch**: `[009-multi-step-recommend]`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "修改逻辑，由于直接上传文件作为prompt可能过长，因为分成几步完成，第一步将用户prompt和上传文件中的"所属学科"一起发送给llm，由于所属学科是符合统一规定的因此可以先以此条件过滤出符合的课程，第二步将课程名称、序号、时间、主讲人、开课地点等非敏感信息和用户prompt传给llm，进一步筛选符合条件（方向、时间等其他约束)，最后生成完整课程表。注意不要每次都将完整文件内容发送给llm，可让llm返回序号并在本地重新查找文件以获取详细信息，llm只需要知道必要的非敏感信息。"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Multi-Step Subject Filtering (Priority: P1)

学生上传课表文件后，输入选课需求（如"推荐计算机课程"）。系统将需求与文件中的"所属学科"字段发送给 LLM，LLM 返回匹配的学科列表，系统据此在本地过滤出对应课程子集。

**Why this priority**: 这是整个多步推荐流程的第一步，也是最关键的数据缩减步骤。当课表包含数千条记录时，先按学科过滤可将数据量缩减 90% 以上，从根本上解决 prompt 过长导致内容安全策略触发或 token 超限的问题。

**Independent Test**: 可通过上传包含多学科课表的文件，输入"推荐计算机课程"，验证系统仅向 LLM 发送了学科列表（而非全部课程数据），并在本地准确筛选出计算机相关课程子集。

**Acceptance Scenarios**:

1. **Given** 学生上传了一份包含 2000+ 条课程记录的课表文件（涵盖计算机、数学、物理、文学等多个学科），**When** 学生输入"推荐计算机课程"，**Then** 系统第一步仅将用户需求和"所属学科"列表发送给 LLM，LLM 返回匹配的学科（如"计算机科学与技术"）。
2. **Given** LLM 返回了匹配的学科列表，**When** 系统在本地执行过滤，**Then** 仅保留属于匹配学科的课程记录（如从 2000 条缩减至约 200 条）。
3. **Given** 用户的需求不匹配任何已知学科（如"推荐创意写作课程"但无此学科），**When** 系统执行学科过滤，**Then** 系统回退到不过滤学科，使用全部课程进行后续推荐。

---

### User Story 2 - Non-Sensitive Detail Filtering (Priority: P2)

在学科过滤后的课程子集上，系统将课程的非敏感信息（名称、序号、上课时间、主讲人、上课地点、学分）和用户需求发送给 LLM，LLM 根据方向偏好、时间约束等进一步筛选并返回推荐方案。

**Why this priority**: 第二步精细筛选是在已缩减的数据集上进行，确保 LLM 能有效分析每门课程的具体属性（时间冲突、方向匹配等），同时避免暴露敏感信息（如学分详情、选课人数等内部数据）。

**Independent Test**: 可通过向 LLM 验证发送的消息中仅包含非敏感字段，且 LLM 返回的推荐方案中课程时间无冲突、方向符合用户描述。

**Acceptance Scenarios**:

1. **Given** 第一步已完成学科过滤，**When** 系统将过滤后的课程非敏感信息（含学分）和用户需求发送给 LLM，**Then** LLM 返回的推荐方案中所有课程属于同一学科方向，且时间互不冲突。
2. **Given** 用户明确指定了时间约束（如"周五不想上课"），**When** LLM 生成推荐方案，**Then** 推荐方案中不包含周五的课程。
3. **Given** LLM 返回了推荐方案，**When** 系统处理推荐结果，**Then** 系统使用 LLM 返回的课程序号在本地重新查找完整的课程详情（包括学分、容量等敏感信息）来组装最终响应。

---

### User Story 3 - End-to-End Recommendation Flow (Priority: P3)

学生完成上传和对话后，获得一份完整的、无冲突的周课程表推荐方案，包含课程名称、时间、地点、教师、学分等完整信息。

**Why this priority**: 端到端体验验证整个多步流程的正确性，确保最终用户看到的是完整的课程推荐方案而非中间过程数据。

**Independent Test**: 可通过完整流程测试——上传课表 → 输入需求 → 验证返回的推荐方案包含完整课程信息且时间无冲突。

**Acceptance Scenarios**:

1. **Given** 学生上传了课表文件并输入了选课需求，**When** 系统完成两步推荐流程，**Then** 返回 1-3 个命名推荐方案，每个方案包含完整课程信息（名称、时间、地点、教师、学分）且时间无冲突。
2. **Given** 系统返回了推荐方案，**When** 学生查看方案详情，**Then** 方案包含匹配度评分和推荐理由。

---

### Edge Cases

- 课表文件中"所属学科"字段为空或格式不统一时，系统应回退到使用全部课程数据
- LLM 在第一步返回了不存在的学科名称时，系统应忽略无效学科并继续处理有效学科
- LLM 在第二步返回的序号在本地查找中不存在时，应跳过该课程并继续处理其他课程
- 用户多次连续发送不同需求时，每一步的中间结果不应影响后续请求的独立性
- 第二步筛选后无课程匹配用户需求时，系统直接返回"无匹配课程"提示，不做额外降级（不回退到全量数据或关键词搜索）

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 将推荐流程分为至少两个独立步骤：学科过滤和详细筛选
- **FR-002**: 系统 MUST 在第一步将用户需求和课表中所有存在的"所属学科"列表发送给 LLM，LLM 从中做多选题返回匹配学科，不得发送完整课程数据
- **FR-003**: 系统 MUST 根据 LLM 返回的学科列表在本地过滤课程数据
- **FR-004**: 系统 MUST 在第二步仅发送非敏感课程信息（课程名称、序号、上课时间、主讲人、上课地点、学分）给 LLM
- **FR-005**: 系统 MUST 使用 LLM 返回的课程序号在本地重新查找完整课程详情
- **FR-006**: 系统 MUST 在学科过滤无结果时回退到使用全部课程数据继续推荐
- **FR-007**: 系统 MUST 对最终推荐方案执行时间冲突检测
- **FR-008**: 系统 MUST 在任一步骤的 LLM 调用失败时提供降级方案（关键词搜索）
- **FR-009**: 系统 MUST 保持单次推荐完成的交互模式，不追问用户信息

### Key Entities

- **Course (课程)**: 课程记录，包含学科、名称、序号、时间、主讲人、地点、学分等属性。其中"所属学科"是第一步过滤的关键字段，序号是第二步返回后本地查找的标识。
- **RecommendationPlan (推荐方案)**: 包含方案名称、课程列表、匹配度评分、推荐理由、时间冲突信息。
- **Subject (所属学科)**: 课程的学科分类字段，用于第一步粗粒度过滤。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 当课表包含 2000+ 条记录时，第一步发送给 LLM 的数据量（学科列表）不超过原始数据的 1%
- **SC-002**: 当课表包含 2000+ 条记录时，第二步发送给 LLM 的课程数量等于学科过滤后的子集，不做额外截断
- **SC-003**: 推荐结果中 100% 的课程时间无冲突
- **SC-004**: 从用户输入需求到看到推荐结果的端到端响应时间不超过 30 秒
- **SC-005**: 推荐方案中课程信息完整率 100%（包含名称、时间、地点、教师、学分所有字段）
- **SC-006**: 因 prompt 过长导致的 LLM 内容安全过滤触发率降至 0%

## Assumptions

- 课表文件中的"所属学科"字段遵循统一的学科分类标准，可直接用于精确过滤
- MiMo API 支持标准的 JSON 结构化输出（response_format: json_object）
- 课表文件已通过现有的导入流程正确解析并存储在 SessionStore 中
- 用户的选课需求以自然语言中文描述为主
- LLM 能正确理解学科名称并返回匹配的学科列表
