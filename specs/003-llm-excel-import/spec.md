# Feature Specification: 学生课表智能导入与推荐

**Feature Branch**: `[003-llm-excel-import]`
**Created**: 2026-04-22
**Status**: Draft
**Input**: User description: "增加功能，上传的excel文件不一定是固定格式，需要通过llm分析，注意检查是否经过llm提取后仍然不符合格式。实现方式：自定义需要的head，将上传的文件的head和自己需要的head同时发送给llm做对应。注意不要发送整个文件给llm以节省token。增加功能：对于上传的文件解析更加鲁棒，例如先提取几行给llm做分析节省token，同时转化head和日期等为统一格式，只对课程名称、分类等做llm分析，讲课人由于是人名可通过算法处理，时间冲突在经过日期统一后也可以通过算法处理，不要通过llm，节省token。增加功能，对于上传的文件的head不做强制要求，当缺少需要的信息时可以降级功能，例如没有时间，可以仅推荐课程名称。没有分类时，可以通过课程名称推断，没有主讲人，就不根据主讲人推荐，但是不要总是报错。"

**澄清记录**:
- 本项目完全由学生上传课表，不存在教师/管理员上传场景。
- 课程数据是一次性的：每次新会话需重新上传 Excel，数据不持久化到数据库。
- 项目本质是 LLM 的 UX 封装，优化推荐结果+课表的界面交互和分享功能。

## User Scenarios & Testing

### User Story 1 - 学生上传任意格式课表（Priority: P1）

学生从学校教务系统导出一份 Excel 课表，列名可能是中文、英文或混合格式。学生上传该文件后，系统通过 LLM 自动识别列含义，提取课程数据并在内存中用于本次会话的推荐。不存储到数据库，下次会话需重新上传。

**Why this priority**: 这是学生使用系统的第一步，也是最大的使用障碍——当前系统要求固定列名。

**Independent Test**: 上传一份中文列名的 Excel，验证系统自动识别列并展示提取到的课程列表，刷新页面后数据消失（不持久化）。

**Acceptance Scenarios**:

1. **Given** 学生上传一份中文列名（课程编号、课程名称、学分...）的 Excel，**When** 文件上传完成，**Then** 系统通过 LLM 识别列映射，提取课程数据并在页面展示课程列表。
2. **Given** 学生上传的 Excel 列顺序不同且包含无关列，**When** 系统处理文件，**Then** 正确映射相关列，忽略无关列。
3. **Given** 本次会话已完成推荐，**When** 学生刷新页面或开始新会话，**Then** 课程数据消失，需重新上传 Excel。

### User Story 2 - Token 节省：仅发送表头+样本行，算法处理确定性字段（Priority: P1）

系统不将整个文件发送给 LLM，仅发送表头和前几行样本数据。日期标准化、人名提取等确定性任务通过算法完成，仅课程名称和分类等语义字段走 LLM。

**Why this priority**: 与 P1 并列——控制 LLM 调用成本是系统可持续运行的基础。

**Independent Test**: 上传一份 100 行的 Excel，验证 LLM 调用 token 数不超过 500，且日期/学期字段未经过 LLM 处理。

**Acceptance Scenarios**:

1. **Given** Excel 学期列为"2026年春"、"2025-2026-2"等格式，**When** 系统预处理，**Then** 通过算法统一为标准格式，不消耗 LLM token。
2. **Given** Excel 教师列为"张三 教授"，**When** 系统预处理，**Then** 通过算法提取纯人名"张三"，不消耗 LLM token。
3. **Given** 仅表头和前 3 行数据发送给 LLM，**When** LLM 返回列映射，**Then** 发送 token 不超过 500。

### User Story 3 - 缺失字段优雅降级（Priority: P1）

学生上传的课表可能缺少某些列。系统不应因缺少非关键信息而拒绝或频繁报错，应以降级模式继续提供推荐。

**Why this priority**: 不同学校导出的课表格式差异很大，缺少部分字段是常态。

**Independent Test**: 上传一份仅有"课程名称"和"学分"两列的 Excel，验证系统成功导入并提供基础推荐。

**Acceptance Scenarios**:

1. **Given** Excel 没有上课时间列，**When** 导入完成，**Then** 课程成功导入，推荐时不进行时间冲突检测，不报错。
2. **Given** Excel 没有分类列，**When** 导入完成，**Then** 系统通过 LLM 根据课程名称推断分类。
3. **Given** Excel 没有主讲人列，**When** 导入完成，**Then** 推荐时不按教师筛选，不报错。
4. **Given** Excel 仅有课程名称一列，**When** 导入完成，**Then** 系统以降级模式提供按名称匹配的基础推荐。
5. **Given** 缺少课程编号列，**When** 导入完成，**Then** 系统用课程名称作为去重依据。

### User Story 4 - 推荐结果+课表可视化与分享（Priority: P2）

学生输入选课需求后，系统返回推荐方案并在可视化课表中展示课程时间分布。学生可保存方案并分享。

**Why this priority**: 这是项目的核心价值——在 LLM 推荐基础上提供更好的界面交互体验。

**Independent Test**: 获取推荐结果后，验证课表正确展示每门课程在对应星期和时间段的位置，冲突高亮。

**Acceptance Scenarios**:

1. **Given** 学生输入"推荐数据结构和算法课程"，**When** 系统返回推荐方案，**Then** 每个方案卡片内展示可视化课表，课程按时间定位，冲突红色高亮。
2. **Given** 推荐方案中课程缺少时间信息，**When** 展示课表，**Then** 该课程在课表中标记为"时间未指定"而非空白。
3. **Given** 学生对推荐方案满意，**When** 点击保存，**Then** 方案保存到个人设备（cookie 关联），可后续查看和导出 PDF。

### User Story 5 - 格式校验与错误反馈（Priority: P2）

系统对提取后的数据进行校验，仅对存在的核心字段报错，不因缺失可选字段阻止导入。

**Independent Test**: 上传一份学分字段为"三"的 Excel，验证校验给出明确错误提示但不因缺少时间列报错。

**Acceptance Scenarios**:

1. **Given** 某行学分字段为"三"，**When** 校验，**Then** 提示"第 X 行：学分必须为数字"。
2. **Given** 某行课程名称为空，**When** 校验，**Then** 提示"第 X 行：课程名称不能为空"。
3. **Given** 文件没有时间列，**When** 校验，**Then** 不产生时间相关的错误提示。

### User Story 6 - 列映射预览（Priority: P3）

LLM 完成识别后展示映射结果和降级说明，学生可确认后再进入推荐。

**Acceptance Scenarios**:

1. **Given** LLM 返回列映射，**When** 展示结果，**Then** 学生看到映射关系和缺失字段的影响说明。
2. **Given** 识别到缺失分类列，**When** 展示结果，**Then** 提示"分类将根据课程名称自动推断"。

### Edge Cases

- 文件不含任何课程相关列，提示"未识别到课程数据，请检查文件内容"。
- 仅有课程名称一列，提供按名称匹配的降级推荐。
- 分类列全部为空，LLM 逐行推断，失败标记为"未知"。
- LLM 超时或失败，回退到固定格式解析。
- 文件行数极少（< 3 行数据），发送所有行给 LLM。
- Excel 包含多个 Sheet，默认读取第一个。
- 新会话开始时旧的课程数据自动清除。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 由学生（非管理员）上传 Excel 课表文件，上传入口对学生可见。
- **FR-002**: 系统 MUST 将上传的课程数据保存在会话内存中，不持久化到数据库，新会话需重新上传。
- **FR-003**: 系统 MUST 维护一份字段定义列表，区分核心字段（课程名称、学分）和可选字段（课程编号、教师、容量、地点、校区、类别、学期、上课时间）。
- **FR-004**: 系统 MUST 仅将表头行和前 N 行样本数据（默认 3 行）发送给 LLM，不发送整个文件。
- **FR-005**: 系统 MUST 将字段定义和上传文件表头同时提供给 LLM 做列映射。
- **FR-006**: 系统 MUST 支持中英文混合列名识别。
- **FR-007**: 系统 MUST 通过算法标准化日期/学期字段和提取教师人名，不调用 LLM。
- **FR-008**: 系统 MUST 仅对课程名称和分类字段调用 LLM 做语义理解。
- **FR-009**: 系统 MUST 对核心字段（课程名称、学分）为空或格式错误的行给出错误提示。
- **FR-010**: 系统 MUST 在缺少可选字段时以降级模式导入，不报错。
- **FR-011**: 系统 MUST 在缺少分类时通过 LLM 根据课程名称推断。
- **FR-012**: 系统 MUST 在缺少上课时间时跳过冲突检测，不报错。
- **FR-013**: 系统 MUST 在缺少主讲人时跳过按教师筛选，不报错。
- **FR-014**: 系统 MUST 在缺少课程编号时用课程名称去重。
- **FR-015**: 系统 MUST 在 LLM 不可用时回退到固定格式解析。
- **FR-016**: 系统 MUST 在导入前展示列映射和降级说明。
- **FR-017**: 系统 MUST 在推荐结果中展示可视化课表（周一至周日），课程按时间定位。
- **FR-018**: 系统 MUST 在课表中用红色高亮标记时间冲突的课程。
- **FR-019**: 系统 MUST 对缺少时间信息的课程在课表中标记为"时间未指定"。
- **FR-020**: 系统 MUST 支持学生保存推荐方案（cookie 关联设备），可后续查看和导出 PDF。

### Key Entities

- **会话课程数据 (Session Course Data)**: 存在于内存中的课程列表，随会话结束而清除。包含课程名称、学分及所有可选字段。
- **字段定义 (Field Schema)**: 每个字段的名称、类型、层级（核心/可选）、别名列表、缺失降级策略。
- **列映射 (Column Mapping)**: LLM 返回的 Excel 列名到系统字段的对应关系。
- **降级报告 (Degradation Report)**: 缺失的可选字段及其对推荐功能的影响说明。
- **推荐方案 (Recommendation Plan)**: 包含方案名称、课程列表、总学分、匹配度、冲突信息、可视化课表。
- **保存方案 (Saved Plan)**: 持久化到数据库的推荐方案快照，通过 cookie device_id 关联。

## Success Criteria

### Measurable Outcomes

- **SC-001**: 学生上传非标准格式 Excel 后，30 秒内完成解析并展示课程列表。
- **SC-002**: LLM 每次列映射调用 token 不超过 500，相比发送整个文件节省 90%+。
- **SC-003**: LLM 列映射准确率不低于 95%。
- **SC-004**: 日期和人名字段 100% 通过算法处理，不消耗 LLM token。
- **SC-005**: 仅含核心字段的 Excel 导入成功率 100%。
- **SC-006**: 缺少可选字段时无错误提示，导入成功率 100%。
- **SC-007**: 缺少分类时 LLM 推断准确率不低于 80%。
- **SC-008**: 推荐结果中课表可视化正确展示课程时间分布，100% 的有时间课程正确定位。
- **SC-009**: 学生可在推荐后保存方案并在新会话中查看已保存方案。

## Assumptions

- 系统已集成 LLM 服务。
- 学生上传的是自己学校的可选课程列表（课表/选课手册）。
- 课程数据按会话在内存中处理，不持久化到 Course 表。
- 上传的 Excel 第一个 Sheet 包含课程数据。
- 文件大小不超过 5MB，行数不超过 2000 行。
- 保存的推荐方案仍需持久化到数据库（SavedPlan 表），关联 cookie device_id。
- 推荐服务在新架构下从内存中的会话数据获取课程列表，而非从数据库查询。
