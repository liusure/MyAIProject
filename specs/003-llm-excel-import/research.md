# Research: 学生课表智能导入与推荐

## 决策 1: 会话内存存储方案

**选择**: 使用内存 dict，以 device_id 为 key 存储课程列表，不写入数据库。

**理由**:
- 用户明确课程数据是一次性的，每次新会话需重新上传
- 避免不同用户数据混合的隐私问题
- 避免数据库膨胀（学生可能多次上传不同文件）
- 简单 dict 结构即可满足需求，无需引入 Redis 等额外依赖

**备选方案**:
- Redis session store: 引入额外基础设施依赖，杀鸡用牛刀
- 数据库临时表: 仍需定时清理，且不同用户数据会混合
- 前端内存: 刷新丢失太快，推荐服务需要访问数据

## 决策 2: LLM 列映射策略

**选择**: 仅发送表头 + 前 3 行样本数据给 LLM，请求返回列映射 JSON。不发送完整文件。

**理由**:
- 用户明确要求节省 token，token 上限 500
- 列映射仅需看到列名和少量样本即可完成
- 完整文件可能导致 token 超限且增加延迟

**实现细节**:
- 先检查列名是否完全匹配（快速路径），匹配则跳过 LLM 调用
- LLM prompt 包含: 系统字段定义列表 + 上传文件表头 + 前 N 行数据
- LLM 返回格式: `{"mappings": [{"source": "原始列名", "target": "系统字段"}], "unmapped": ["未匹配列"]}`
- LLM 超时或失败时回退到固定格式解析（复用现有 ImportParser）

**备选方案**:
- 全走 LLM 解析每行数据: token 成本过高
- 纯算法模糊匹配: 无法处理语义差异（如"任课教师"→"instructor"）
- Embedding 相似度: 增加了 embedding 模型依赖

## 决策 3: 字段处理分工

**选择**: 混合处理——LLM 只处理语义字段，算法处理确定性字段。

| 字段 | 处理方式 | 理由 |
|------|----------|------|
| 列名映射 | LLM | 列名语义差异大，需理解 |
| 课程名称 | 直接透传 | 已映射，无需额外处理 |
| 分类推断 | LLM | 缺失时需根据课程名推断语义 |
| 日期/学期 | 算法正则 | 格式有限，可穷举处理 |
| 教师人名 | 算法提取 | 去掉"教授"等后缀即可 |
| 学分 | 算法转换 | 中文数字转阿拉伯数字 |
| 时间冲突 | 算法 | 已标准化后可直接比较 |

**理由**:
- 确定性字段用 LLM 是浪费——日期格式有限，人名规则明确
- 用户明确要求"讲课人由于是人名可通过算法处理，时间冲突在经过日期统一后也可以通过算法处理，不要通过llm，节省token"

## 决策 4: 降级策略设计

**选择**: 字段分为核心/可选两层，核心字段缺失报错，可选字段缺失降级。

**核心字段** (缺失不可用):
- 课程名称 (name): 推荐的基础依据
- 学分 (credit): 方案匹配的必需条件

**可选字段** (缺失降级):
- 课程编号 (course_no) → 用名称去重
- 分类 (category) → LLM 根据名称推断
- 上课时间 (schedule) → 跳过冲突检测，课表标记"时间未指定"
- 主讲人 (instructor) → 跳过教师筛选
- 容量/地点/校区/学期 → 跳过相关筛选

**理由**:
- 不同学校导出格式差异很大，缺少部分字段是常态
- 强制所有字段会导致大部分学生无法使用
- 分类缺失时 LLM 推断是合理的——课程名称本身就包含分类语义

## 决策 5: 冲突检测接入

**选择**: 将现有 ConflictEngine 接入推荐流程，不再返回空 conflicts。

**当前状态**: ConflictEngine 已完整实现（时间/先修/通勤三类检测），但 RecommendService 从未调用。

**变更**:
- RecommendService 构造函数接受 session_courses 时，同时将课程转为 dict 格式传给 ConflictEngine
- 缺少 schedule 字段的课程自动跳过时间检测（ConflictEngine 已支持空 schedule）
- 先修课检测在 session 数据模式下不适用（无 DB 中的先修关系），跳过

**理由**:
- 冲突检测是核心功能需求（FR-012, FR-018）
- 现有代码已完整，只需接入，不需要重写
