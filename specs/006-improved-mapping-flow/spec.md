# Feature Specification: 映射流程优化

**Feature Branch**: `006-improved-mapping-flow`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "选择未识别列的对应关系时，左侧为你需要的head，右侧下拉为未识别列，并且在中间输出此列用途以及不选择会降级何种功能。在确认对应关系选择导入后，应给与返回按钮，给予用户重新选择对应关系的机会。总之不能让整体的流程没有回退的机会。"

## User Scenarios & Testing

### User Story 1 - 系统需求驱动的映射选择 (Priority: P1)

学生上传 Excel 后进入映射预览页面。对于系统需要但未被 LLM 识别的字段，页面以「系统需要什么」为视角展示：左侧显示系统字段名称（如"上课时间"），中间显示该字段的用途以及不映射时功能降级说明，右侧提供下拉框让学生从未识别的文件列中选择对应的列。

**Why this priority**: 当前界面以「未识别列」为视角，学生不理解为什么要映射、不映射会怎样。改为以系统需求驱动，配合降级说明，让学生做出知情选择，显著提升可用性和数据完整性。

**Independent Test**: 上传一个包含多个未识别列的 Excel，查看映射预览页面，确认每个未映射的系统字段左侧显示字段名、中间显示用途和降级说明、右侧显示下拉框可选择文件列。

**Acceptance Scenarios**:

1. **Given** LLM 识别了"课程名称→name"和"学分→credit"，但未识别"上课时间"和"上课地点"列，**When** 用户查看映射预览页面，**Then** 页面显示"上课时间"和"上课地点"两个系统字段行，每行左侧为字段名、中间为用途+降级说明、右侧为下拉框可选择未识别列
2. **Given** 用户在"上课时间"行的下拉框中选择了文件中的"开课时间"列，**When** 用户查看其他系统字段行的下拉框，**Then** "开课时间"不再出现在其他下拉选项中（已被占用）
3. **Given** 某系统字段的所有未识别列都已被其他字段选走，**When** 用户查看该字段的下拉框，**Then** 下拉框仅显示"不映射"选项，中间仍展示降级说明

---

### User Story 2 - 导入后可回退重新选择 (Priority: P1)

学生确认映射关系并导入后，如果发现数据有问题或想重新调整映射，应能通过返回按钮回到映射预览页面重新选择对应关系。

**Why this priority**: 当前流程一旦确认导入就无法回退，学生如果发现映射选错只能重新上传文件。提供返回能力是基本的用户体验保障。

**Independent Test**: 完成一次导入后，在导入结果页面点击返回按钮，验证页面回到映射预览状态，且之前的选择被保留可修改。

**Acceptance Scenarios**:

1. **Given** 用户已完成导入并看到"成功导入 X 门课程"，**When** 用户点击返回按钮，**Then** 页面回到映射预览状态，之前的 LLM 映射和手动选择均被保留
2. **Given** 用户在返回映射预览后修改了某个映射选择，**When** 用户再次点击确认导入，**Then** 导入使用新的映射关系覆盖之前的结果
3. **Given** 用户在映射预览页面，**When** 用户点击返回按钮，**Then** 页面回到文件上传初始状态（可重新选择文件）

---

### User Story 3 - 整体流程可回退 (Priority: P2)

整个导入流程的每一步都有返回路径，用户不会被卡在任何一步。

**Why this priority**: 流程不可回退是严重的可用性缺陷，用户可能因误操作无法挽回。

**Independent Test**: 在流程的每个步骤（上传→预览→导入结果）都尝试返回操作，确认每一步都能回到上一步。

**Acceptance Scenarios**:

1. **Given** 用户在映射预览页面，**When** 用户点击"重新选择文件"按钮，**Then** 回到文件上传初始状态
2. **Given** 用户在导入结果页面，**When** 用户点击"返回修改映射"按钮，**Then** 回到映射预览页面
3. **Given** 用户在导入结果页面，**When** 用户点击"重新上传"按钮，**Then** 回到文件上传初始状态

---

### Edge Cases

- 用户返回映射预览后不做任何修改直接再次确认导入 → 应正常导入，结果不变
- 用户返回映射预览后取消之前手动选择的映射（改为"不映射"）→ 对应列从映射中移除，相关数据不再导入
- 导入结果页面的返回按钮和重新上传按钮应明确区分功能差异（一个回预览保留选择，一个完全重来）
- LLM 成功映射所有字段时，手动映射区域不展示，用户仍可直接确认导入或重新选择文件

## Clarifications

### Session 2026-04-23

- Q: LLM未映射必填字段时如何处理？ → A: 必填字段(name/credit)也展示在映射UI中，用红色高亮表示其重要性，但不强制阻断导入流程。LLM本身具有鲁棒性，不需要报错阻断。
- Q: 已被LLM映射的字段是否可修改？ → A: 仅展示未映射的系统字段供手动选择。LLM已映射的字段在上方只读表格中展示，不可修改。
- Q: 返回映射预览后重新导入的数据处理方式？ → A: 完全替换——每次重新导入都清空旧课程数据，用新映射结果重新导入全部课程。
- Q: LLM完美映射所有字段时界面如何展示？ → A: 仅展示只读映射表格和"所有字段已成功识别"成功提示，隐藏手动映射区域。"确认导入"和"重新选择文件"按钮仍然可用。

## Requirements

### Functional Requirements

- **FR-001**: 系统 MUST 在映射预览页面为每个未映射的系统字段展示一行（包括必填字段 name、credit），左侧为系统字段中文名称，中间为字段用途说明和不映射时的降级影响描述，右侧为下拉框可从未识别的文件列中选择。必填字段（name、credit）用红色高亮标注以提示重要性
- **FR-002**: 下拉框选项 MUST 包含所有尚未被映射（LLM映射+用户手动映射）的文件列，以及一个"不映射"的默认选项
- **FR-003**: 用户选择后，系统 MUST 将该映射关系合并到最终映射结果中，与 LLM 自动识别的映射一起参与数据导入
- **FR-004**: 用户未选择（保持"不映射"默认值）的系统字段 MUST 显示降级说明，让用户知晓不映射的后果
- **FR-005**: 系统 MUST 在导入结果页面提供"返回修改映射"按钮，点击后回到映射预览状态，保留之前的映射选择
- **FR-006**: 系统 MUST 在映射预览页面提供"重新选择文件"按钮，点击后回到文件上传初始状态
- **FR-007**: 用户从导入结果返回映射预览后重新确认导入时，系统 MUST 用新的映射结果覆盖之前的导入数据
- **FR-008**: 当 LLM 成功映射所有系统字段时，系统 MUST 隐藏手动映射区域，仅展示只读映射表格和成功提示信息

### Key Entities

- **SystemFieldMapping**: 系统字段映射关系，包含系统字段key、中文名称、用途说明、降级影响、用户选择的文件列（可为空）
- **MappingResult**（已有）: 最终映射结果 = LLM自动映射 + 用户手动映射

## Success Criteria

### Measurable Outcomes

- **SC-001**: 用户能够为 100% 的未映射系统字段看到用途说明和降级影响描述
- **SC-002**: 用户从未识别列中选择映射的操作步骤数 ≤ 2步（点击下拉→选择列）
- **SC-003**: 用户在任意流程步骤都可通过 ≤ 1次点击返回上一步
- **SC-004**: 用户返回映射预览后重新导入，新映射覆盖旧数据的响应时间 ≤ 3秒

## Assumptions

- 系统字段列表固定为 10 个，其中 name（课程名称）和 credit（学分为必填字段），其余 8 个为可选字段
- 必填字段（name、credit）在映射UI中以红色高亮显示，但不阻断导入流程——LLM本身具有鲁棒性，不需要报错阻断
- 字段用途说明和降级影响描述使用后端 DEGRADATION_IMPACTS 数据，前端从 analyze 接口获取
- 返回映射预览时保留用户的 session 数据（服务端内存中的 raw_data 和原始映射结果）
- 导入结果页面的"返回修改映射"和"重新上传"是两个独立操作，功能不同
