# Feature Specification: 课程数据仅存会话，推荐历史持久化

**Feature Branch**: `[008-session-only-courses]`
**Created**: 2026-04-23
**Status**: Draft
**Input**: User description: "数据库只保存推荐历史，不保存课程记录本身，每次都重新上传课程文件"

## Clarifications

### Session 2026-04-23

- Q: 推荐历史中课程数据的存储粒度？ → A: 快照保存 — 推荐结果中的完整课程信息（名称、学分、教师、时间安排等）随推荐方案一起持久化，历史推荐始终可完整查看

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 上传课程后推荐，刷新后需重新上传 (Priority: P1)

用户上传 Excel 课程文件后，使用推荐功能获得推荐方案。推荐历史（对话记录和推荐结果）被保存到数据库。用户刷新页面后，课程数据消失（因为仅存在内存中），需要重新上传文件才能再次使用推荐功能。

**Why this priority**: 这是核心数据流 — 明确课程数据和推荐历史的生命周期边界。确保用户理解"每次都要上传"的行为。

**Independent Test**: 上传课程文件 → 获得推荐 → 刷新页面 → 确认课程列表为空，推荐历史仍在 → 重新上传文件 → 再次获得推荐。

**Acceptance Scenarios**:

1. **Given** 用户已上传课程文件且获得推荐，**When** 用户刷新页面，**Then** 课程列表为空，推荐历史（对话记录）仍可查看
2. **Given** 用户刷新页面后课程列表为空，**When** 用户尝试发起推荐，**Then** 系统提示"请先上传课表文件"
3. **Given** 用户重新上传课程文件，**When** 用户发起推荐，**Then** 系统基于新上传的课程数据返回推荐方案

---

### User Story 2 - 推荐历史持久化 (Priority: P1)

用户每次获得推荐后，推荐方案和对话记录被保存到数据库。用户可以查看历史推荐记录，即使课程数据已过期。

**Why this priority**: 推荐历史是用户的核心资产，需要持久化以供回顾和比较。

**Independent Test**: 上传课程 → 获得推荐 → 刷新页面 → 确认历史对话和推荐方案仍可查看。

**Acceptance Scenarios**:

1. **Given** 用户已获得多次推荐，**When** 用户刷新页面，**Then** 对话历史完整保留，包含之前的推荐方案
2. **Given** 用户查看历史推荐记录，**When** 会话中已无课程数据，**Then** 系统完整展示推荐方案（方案名、匹配度、学分）和每个课程的详细信息（含周课程表），因为这些信息已随推荐结果快照保存

---

### User Story 3 - 服务端不再回退到数据库课程 (Priority: P2)

当用户未上传课程文件时，推荐服务不再从数据库查询课程作为回退，而是直接提示用户上传文件。

**Why this priority**: 消除当前架构中的歧义 — 数据库中的课程数据可能是旧数据或测试数据，不应作为推荐来源。

**Independent Test**: 不上传课程文件直接请求推荐 → 确认系统提示"请先上传课表文件"而非返回数据库中的旧课程推荐。

**Acceptance Scenarios**:

1. **Given** 用户未上传课程文件且会话中无课程数据，**When** 用户输入推荐请求，**Then** 系统返回"请先上传课表文件"提示，不返回任何推荐方案
2. **Given** 数据库中存在课程记录（来自旧版本），**When** 用户未上传文件就请求推荐，**Then** 系统不会使用数据库中的课程数据

---

### Edge Cases

- 用户上传文件后，在获得推荐前关闭浏览器，再次打开时课程数据丢失，推荐功能如何引导？
- 推荐历史中的课程名称与新上传文件中的课程名称相同但详情不同（如时间变更），如何处理？
- 多个用户同时使用，会话隔离是否可靠？

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 课程数据 MUST 仅存储在内存会话中，不持久化到数据库
- **FR-002**: 用户刷新页面或会话过期后，课程数据 MUST 自动消失
- **FR-003**: 推荐历史（对话记录和推荐方案）MUST 持久化到数据库，方案中的课程信息以快照形式完整保存（包含课程名称、学分、教师、时间安排、地点等）
- **FR-004**: 当会话中无课程数据时，推荐服务 MUST 返回"请先上传课表文件"提示，不从数据库回退
- **FR-005**: 数据库中的 courses 表（如有）MUST 不再作为推荐数据来源
- **FR-006**: 用户每次使用推荐功能前 MUST 需要先上传课程文件

### Key Entities *(include if feature involves data)*

- **SessionCourse（会话课程）**: 仅存在于内存中的课程数据，生命周期与浏览器会话一致。包含课程名称、学分、教师、时间安排等。
- **Conversation（对话记录）**: 持久化到数据库的对话历史，包含用户消息和系统回复。
- **RecommendationHistory（推荐历史）**: 对话记录中的推荐方案部分，持久化到数据库。每个方案中的课程信息以快照形式保存完整详情（名称、学分、教师、周课程表等），不依赖会话数据即可查看。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 用户刷新页面后，课程列表为空，推荐历史仍可查看（100% 场景）
- **SC-002**: 未上传课程文件时，推荐请求 100% 返回"请先上传课表文件"提示
- **SC-003**: 上传课程文件后，推荐功能基于会话课程数据返回结果的成功率 > 95%
- **SC-004**: 重新上传课程文件后，推荐功能在 5 秒内返回结果

## Assumptions

- 用户每次使用推荐功能前都会上传课程文件（这是预期使用模式）
- 推荐历史中的课程信息以名称和基本属性存储，不依赖持久化的课程 ID
- 浏览器会话的生命周期取决于 uvicorn 进程运行时间（服务重启会清除所有会话数据）
- 数据库（PostgreSQL）和缓存（Redis）在生产环境中持续运行
